## What Went Well

1. BMAD 6.8 became operational rather than theoretical.

   The repo moved from mixed migration state into an executable BMAD loop: create story, implement, code-review, patch, validate, PR, merge. Story records now carry baseline commits, Dev Agent Records, review findings, and verification logs.

2. The project now has a real mobile app path.

   Story 1.2 created `apps/mobile` inside the WSL monorepo and established React/Vite tests, API client conventions, CSS structure, auth restore, and mobile build checks. Stories 1.4 and 1.5 reused that shape instead of creating parallel UI patterns.

3. OpenAPI as SSOT held up.

   Every contract story updated `docs/api/openapi.yaml` and regenerated `packages/types/src/api-types.ts`. Review findings repeatedly caught contract drift, and the process improved by Story 1.5.

4. Review gates found real issues before merge.

   Across Epic 1, review found and resolved 57 patch findings. These were not cosmetic: examples included OTP challenge validation, opaque sessions, SSE ownership, idempotent ingest persistence, candidate sanitization, auth restore, account action confirmation, and BYOK envelope retention.

5. Focused probes gave the backend a useful safety net.

   The server now has narrow probes for auth, ingest, Home/Library, and Settings. They run without needing the whole product surface to be finished and caught several boundary issues quickly.

6. Privacy guardrails were consistently strengthened.

   Analytics sanitization expanded from auth PII to XHS URLs, candidate internals, BYOK material, feedback text, URLs, phone/session/cookie/OTP fields, and internal scores.

## Challenges

1. First implementation passes often missed negative paths.

   The recurring review pattern was not "feature missing"; it was "happy path exists, but edge behavior is unsafe or misleading." Examples: invalid OTP/session states, stale login state, SSE ownership, malformed Library filters, BYOK save errors, and feedback fallback semantics.

2. Compatibility aliases increased surface area.

   Auth accepted `otp`/`code`, ingest preserved canonical and deprecated paths, BYOK preserved `/user-key` plus `/byok/*` aliases. This helped migration safety, but it repeatedly created contract and validation drift that tests had to pin down.

3. No-DB/local fallback paths are useful but easy to over-trust.

   Ingest and Library support deterministic no-DB probes, which is valuable for local development. The risk is treating those in-memory paths as equivalent to production persistence. Epic 2 should keep test seams explicit and avoid building product assumptions on local fallback behavior.

4. UI fallback copy needs review as carefully as APIs.

   Story 1.5 showed that text can become a product bug: saying a browser opened or feedback was recorded when neither happened is a trust problem. Copy must reflect actual state transitions, especially around fallback and failure.

5. Browser/dev-server verification was not consistently available.

   Story 1.2 encountered local listening restrictions before approval was configured. Later stories had strong unit/build coverage, but browser-level inspection remains a useful additional gate for UI-heavy work.

## Key Lessons

1. Treat review findings as workflow signal, not cleanup noise.

   The review phase consistently surfaced gaps in ownership, privacy, idempotency, and fallback behavior. Epic 2 should assume this phase is part of implementation, not optional polish.

2. Every new route needs three companions.

   A route is not "done" until it has OpenAPI shape, generated types, and a focused probe or UI test. Stories that followed this rule were easier to stabilize.

3. User-scoped data access must be tested explicitly.

   SSE events, Library reads, candidates, BYOK status, and account queues all rely on authenticated ownership. The safest pattern is to test cross-user denial in the same story that introduces the route.

4. Deterministic local behavior is a product accelerator.

   Stubs for OTP, captcha, ingest adapters, and no-DB stores made the project runnable and testable in WSL. The rule is to keep stubs named, scoped, and covered so they do not masquerade as final integrations.

5. UI trust surfaces require strict failure honesty.

   Login, ingest, Home, Settings, BYOK, feedback, and account deletion are all trust-bearing. They should never fake success, hide operational failures, or leak sensitive material into analytics.

## Technical Debt and Deferred Scope

These are acceptable carry-forward items, not failures of Epic 1:

- Production SMS, OAuth/Authing/Jiguang, Tencent captcha, and persistent auth sessions remain outside the current stubbed auth boundary.
- XHS downloader, COS, multimodal extraction, and AMap are adapter seams with deterministic stubs unless environment integrations are configured.
- BYOK has envelope handling and key references, but full durable BYOK lifecycle and provider routing belong to Epic 3.3.
- Feedback has official product-link fallback and a system-mail fallback path, but full internal fallback form persistence, screenshot upload, webhooks, and feedback data API belong to Epic 3.3.
- Account export/delete currently queue entry-point tasks; full file cleanup, telemetry anonymization, and completion state belong to Epic 3.3.
- The current mobile app is a mobile Web MVP. Native WebView behavior should be validated when/if a native wrapper is introduced.
